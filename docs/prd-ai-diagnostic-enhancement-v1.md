### 1.2 核心发现

1. **自然语言 + LLM 专项诊断是差异化优势**：商业产品（Datadog、New Relic）的 AI 助手偏通用运维，非数据库专家级别。本系统在 "LLM 驱动的 MySQL 专项诊断" 赛道有独特定位。
2. **阿里云 DAS 是功能标杆**：从异常检测→自动索引→SQL 限流→弹性伸缩，覆盖 AI4DB 全场景。
3. **从 "诊断建议" 到 "可执行优化" 是关键跃迁**：当前仅输出文本建议，用户需手动执行。业界趋势是提供可直接执行的优化方案。
4. **诊断深度和上下文丰富度决定 AI 分析质量**：当前采集的 MySQL 上下文（5 条慢查询、10 条配置、5 条等待事件）偏少，限制了 AI 分析的精准度。

## 二、功能增强规划

按 **优先级** 分为三个阶段：P0（核心增强）、P1（重要扩展）、P2（长期演进）。

---

### P0 — 核心功能增强（建议优先实施）

#### 2.1 深度上下文采集增强

**现状问题**：当前 `context_builder.py` 采集的数据有限（慢查询 Top 5、配置 10 条、等待事件 5 条），导致 AI 分析缺乏足够信息，诊断精度受限。

**需求描述**：

| 增强项 | 当前 | 目标 | 说明 |
|-------|------|------|------|
| InnoDB 状态 | ❌ | `SHOW ENGINE INNODB STATUS` 解析 | 提取事务、死锁、Buffer Pool、I/O 等关键段落 |
| 进程列表 | ❌ | `SHOW PROCESSLIST` + `performance_schema.threads` | 活跃会话、长事务、阻塞链 |
| 表空间信息 | ❌ | `information_schema.TABLES` 聚合 | 各库/表数据量、碎片率 |
| 复制状态 | ❌ | `SHOW REPLICA STATUS` | 主从延迟、IO/SQL 线程状态 |
| 内存分配 | ❌ | `performance_schema.memory_summary_global_by_event_name` | 内存使用分布 |
| 慢查询数量上限 | 5 条 | 可配置，默认 20 条 | 更多样本提升 AI 分析准确率 |
| 配置变量数量 | 10 条 | 按类别分组采集（InnoDB/连接/日志/复制） | 覆盖 50+ 关键变量 |

**验收标准**：
- AI 诊断回答中能引用 InnoDB 状态、进程列表等深度数据
- 上下文采集耗时不超过 3 秒（并行采集）
- 支持配置采集深度（快速 / 标准 / 深度）

---

#### 2.2 SQL 优化结果结构化与可执行化

**现状问题**：SQL 优化结果以纯文本/Markdown 呈现，用户需要手动复制 SQL 执行。索引建议也是纯文本描述，无法直接生成 DDL。

**需求描述**：

1. **结构化优化输出**：
   - 原始 SQL vs 优化 SQL 的 diff 对比视图
   - 优化前后 EXPLAIN 对比表格（自动执行 EXPLAIN 对比）
   - 预估性能提升百分比（基于 EXPLAIN rows/cost 变化）

2. **可执行索引建议**：
   - AI 输出的索引建议自动生成 `CREATE INDEX` DDL
   - 前端展示 DDL 并提供 "一键复制" 和 "执行预览"（显示预估影响）
   - 标注索引创建的预估耗时和锁影响

3. **优化方案对比**：
   - 当 AI 给出多个优化方案时，以表格展示各方案的预估效果
   - 支持用户选择方案后查看详情

**验收标准**：
- 索引建议 100% 自动生成可执行 DDL
- EXPLAIN 对比展示原始/优化后的 rows、type、key 等关键差异
- 用户可一键复制优化后的 SQL 和 DDL

---

#### 2.3 诊断会话历史持久化

**现状问题**：诊断会话仅存在于前端内存，刷新页面即丢失。无法回溯历史诊断记录。

**需求描述**：

1. **会话持久化**：
   - 新建数据库模型 `DiagnosisSession`（会话 ID、连接 ID、类型、创建时间）
   - 新建模型 `DiagnosisMessage`（会话 ID、角色、内容、上下文快照、时间戳）
   - 支持查看历史会话列表，按连接/时间/类型筛选

2. **会话管理**：
   - 侧边栏展示历史会话列表（参考 ChatGPT 侧边栏交互）
   - 支持继续历史会话（加载上下文）
   - 支持删除、归档会话
   - 支持为会话添加标签（如 "已解决"、"待跟进"）

3. **诊断快照**：
   - 每次诊断自动保存当时的 MySQL 上下文快照
   - 支持对比不同时间点的上下文变化（前后快照 diff）

**验收标准**：
- 刷新页面后历史会话完整保留
- 支持至少保存最近 100 条会话记录
- 会话列表加载时间 < 500ms

---

#### 2.4 AI 诊断报告生成

**现状问题**：AI 分析结果只能在线查看，无法导出。无法定期生成健康报告。

**需求描述**：

1. **单次诊断导出**：
   - 支持将 AI 诊断对话导出为 Markdown / PDF 报告
   - 报告包含：问题描述、AI 分析结论、MySQL 上下文数据、优化建议

2. **健康巡检报告**：
   - 一键触发全面健康检查（组合所有 8 个快速诊断 + 综合分析）
   - 生成结构化报告：健康评分（0-100）、各维度评分、Top 问题列表、优化建议优先级排序
   - 支持定时巡检（每日/每周/自定义周期）

3. **报告模板**：
   - 内置日常巡检、性能优化专项、上线前检查等报告模板
   - 报告结论以 "严重/警告/提示" 三级呈现

**验收标准**：
- 报告可导出为 Markdown 文件
- 健康评分算法基于 AI 分析结果 + 量化指标
- 巡检报告涵盖所有 8 个诊断维度

### P1 — 重要功能扩展

#### 2.5 智能参数调优建议

**现状问题**：当前 AI 可以分析配置问题，但未提供具体的参数调优建议和量化推荐值。

**需求描述**：

1. **参数调优 Agent**：
   - 采集当前 MySQL 全部关键参数（50+ 个）和运行状态
   - AI 分析当前负载特征（OLTP/OLAP/混合），给出参数调优建议
   - 每条建议包含：参数名、当前值、推荐值、修改原因、风险等级、是否需要重启

2. **调优模拟**：
   - 显示参数修改前后的预估影响（如 Buffer Pool 调大后预估命中率变化）
   - 生成 `SET GLOBAL` 语句（动态参数）和 `my.cnf` 配置片段（静态参数）

3. **参数关联分析**：
   - 提示参数之间的依赖关系（如 `innodb_buffer_pool_size` 与 `innodb_buffer_pool_instances`）
   - 警告互相冲突的参数设置

**验收标准**：
- 覆盖 InnoDB、连接、日志、复制四大类参数
- 每条建议包含量化推荐值和依据
- 动态参数生成可执行的 SET 语句

---

#### 2.6 深度索引分析与推荐

**现状问题**：当前索引建议依赖 AI 的通用知识，缺乏对实际工作负载的深度分析。

**需求描述**：

1. **工作负载感知的索引分析**：
   - 采集 `performance_schema.table_io_waits_summary_by_index_usage` 分析索引使用情况
   - 识别：未使用索引、冗余索引（被其他索引覆盖）、缺失索引
   - 计算索引效益比（查询次数 / 更新次数）

2. **索引优化报告**：
   - 展示所有表的索引健康度评分
   - 推荐删除的无用索引（附带 `DROP INDEX` DDL）
   - 推荐新建的索引（附带 `CREATE INDEX` DDL + 预估大小）

3. **复合索引优化**：
   - 分析查询模式，推荐最优的复合索引列顺序
   - 识别可合并为复合索引的多个单列索引

**验收标准**：
- 索引分析覆盖所有用户表
- 冗余索引检测准确率 > 90%
- 生成可直接执行的 DDL 语句

---

#### 2.7 LLM Agent 工具调用（Function Calling）

**现状问题**：当前 AI 只能被动接收上下文数据进行分析，无法主动查询数据库获取需要的信息。

**需求描述**：

1. **Agent 模式**：
   - LLM 可通过 Function Calling / Tool Use 主动调用数据库查询
   - 预定义工具集：
     - `execute_query(sql)` — 执行只读查询（SELECT/SHOW/EXPLAIN）
     - `get_table_structure(table)` — 获取表结构
     - `get_table_indexes(table)` — 获取索引信息
     - `get_process_list()` — 获取进程列表
     - `get_innodb_status()` — 获取 InnoDB 状态
     - `get_variable(name)` — 获取系统变量
   - 所有工具调用仅限只读操作，禁止 DML/DDL

2. **交互式诊断**：
   - AI 在分析过程中可追问更多数据（多轮工具调用）
   - 前端展示 AI 的"思考过程"——调用了哪些工具、获取了什么数据
   - 用户可以看到完整的诊断链路

3. **安全控制**：
   - 工具调用白名单机制
   - 查询超时限制（默认 5 秒）
   - 结果集大小限制（默认 100 行）
   - 审计日志记录所有工具调用

**验收标准**：
- 支持至少 6 种预定义工具
- AI 可在单次对话中多次调用工具
- 所有工具调用有审计日志

---

#### 2.8 多连接对比分析

**现状问题**：当前诊断只能针对单个 MySQL 连接，无法对比多个实例的配置和性能差异。

**需求描述**：

1. **实例对比**：
   - 选择 2-3 个 MySQL 连接进行对比分析
   - 对比维度：配置参数差异、性能指标差异、索引策略差异
   - AI 分析配置差异的影响和优化建议

2. **主从对比**：
   - 主从实例配置一致性检查
   - 主从延迟原因 AI 分析
   - 从库性能优化建议（与主库负载特征差异化配置）

**验收标准**：
- 支持同时对比 2-3 个连接
- 配置差异以 diff 视图呈现
- AI 给出差异化的优化建议

### P2 — 长期演进方向

#### 2.9 MySQL 运维知识库（RAG 增强）

**需求描述**：

1. **知识库构建**：
   - 内置 MySQL 官方文档关键章节（InnoDB、优化器、性能调优）
   - 支持用户上传自定义运维文档、故障手册
   - 向量化存储，支持语义检索

2. **RAG 增强诊断**：
   - AI 诊断时自动检索相关知识库文档
   - 诊断结论附带知识库引用（可追溯依据）
   - 降低 LLM 幻觉风险

**价值**：参考清华 DB-GPT 的知识抽取方案，让 AI 诊断基于可靠的专业知识而非纯模型能力。

---

#### 2.10 异常基线与智能告警

**需求描述**：

1. **性能基线学习**：
   - 持续采集关键指标（QPS、TPS、连接数、Buffer Pool 命中率、慢查询数）
   - 建立时序基线（区分工作日/周末、高峰/低谷）
   - 偏离基线时触发 AI 诊断

2. **智能告警**：
   - 基于基线偏离而非固定阈值的告警策略
   - 告警触发时自动执行 AI 根因分析
   - 告警聚合与降噪（避免告警风暴）

**价值**：参考 SolarWinds DPA 的 ML 基线和 Datadog Watchdog 的自动异常发现。

---

#### 2.11 慢查询智能归类与趋势分析

**需求描述**：

1. **查询指纹聚合**：
   - 对慢查询进行指纹化（参数归一化）
   - 按指纹聚合统计：执行次数、平均耗时、总耗时占比
   - 展示 Top N 查询指纹的趋势变化

2. **AI 趋势分析**：
   - 检测新出现的慢查询（历史未见的指纹）
   - 检测性能退化的查询（执行时间显著增长）
   - AI 分析退化原因（数据量增长？执行计划变化？索引失效？）

**价值**：参考 Percona PMM QAN 的指纹聚合 + SolarWinds DPA 的趋势分析。

---

#### 2.12 优化效果验证闭环

**需求描述**：

1. **优化前后对比**：
   - 用户执行优化建议后，系统自动对比优化前后的性能指标
   - 展示：执行时间变化、扫描行数变化、资源消耗变化

2. **优化反馈学习**：
   - 用户可对 AI 建议评分（有用/无用）
   - 记录优化历史，建立"优化案例库"
   - 后续诊断可引用历史成功案例

**价值**：形成 "诊断 → 建议 → 执行 → 验证 → 反馈" 的完整闭环，参考阿里云 DAS 的自治理念。

## 三、技术架构影响评估

### 3.1 新增数据模型

```
DiagnosisSession      — 诊断会话（id, connection_id, type, title, created_at）
DiagnosisMessage      — 会话消息（session_id, role, content, context_snapshot, created_at）
DiagnosisReport       — 诊断报告（id, connection_id, type, score, content, created_at）
HealthCheckSchedule   — 巡检计划（id, connection_id, cron_expr, enabled）
```

### 3.2 新增/改造服务

| 服务 | 类型 | 说明 |
|------|------|------|
| `context_builder.py` | 改造 | 增加 InnoDB 状态、进程列表等深度采集 |
| `diagnosis_history.py` | 新增 | 会话和消息的 CRUD |
| `report_generator.py` | 新增 | 报告生成与导出 |
| `parameter_tuner.py` | 新增 | 参数调优分析 |
| `index_analyzer.py` | 新增 | 深度索引分析 |
| `agent_tools.py` | 新增 | LLM Agent 工具定义与执行 |
| `prompts.py` | 改造 | 新增参数调优、索引分析等提示词模板 |

### 3.3 新增 API 端点

```
POST   /api/v1/ai/sessions                     — 创建诊断会话
GET    /api/v1/ai/sessions                      — 会话列表
GET    /api/v1/ai/sessions/{id}                 — 会话详情（含消息）
DELETE /api/v1/ai/sessions/{id}                 — 删除会话
POST   /api/v1/ai/report/health-check           — 健康巡检
POST   /api/v1/ai/report/export                 — 导出报告
POST   /api/v1/ai/parameter-tuning/stream       — 参数调优分析（SSE）
POST   /api/v1/ai/index-analysis/stream          — 索引分析（SSE）
POST   /api/v1/ai/compare                       — 多连接对比分析
POST   /api/v1/ai/agent-chat/stream              — Agent 模式对话（SSE）
```

## 四、实施建议

### 4.1 推荐实施顺序

```
第一阶段（P0）—— 夯实核心竞争力
  ├─ 2.1 深度上下文采集增强        → 直接提升现有 AI 诊断质量
  ├─ 2.2 SQL 优化结构化与可执行化  → 提升用户体验和实用性
  ├─ 2.3 诊断会话历史持久化        → 补齐基础体验短板
  └─ 2.4 AI 诊断报告生成          → 满足企业级使用场景

第二阶段（P1）—— 建立差异化优势
  ├─ 2.5 智能参数调优建议          → 高价值功能，业界对标阿里云 DAS
  ├─ 2.6 深度索引分析与推荐        → 高频需求，配合 SQL 优化形成闭环
  ├─ 2.7 LLM Agent 工具调用       → 技术亮点，对标清华 DB-GPT
  └─ 2.8 多连接对比分析            → 企业多实例管理需求

第三阶段（P2）—— 长期技术储备
  ├─ 2.9 知识库 RAG 增强           → 提升诊断可靠性
  ├─ 2.10 异常基线与智能告警       → 从被动诊断到主动发现
  ├─ 2.11 慢查询趋势分析           → 历史维度洞察
  └─ 2.12 优化效果验证闭环         → 完整自治循环
```

### 4.2 关键成功指标

| 指标 | 当前 | P0 完成后目标 | P1 完成后目标 |
|------|------|-------------|-------------|
| AI 诊断精准度（用户满意率） | 未量化 | ≥ 70% | ≥ 85% |
| 单次诊断上下文数据量 | ~15 条 | ~100 条 | ~200 条 |
| 索引建议可执行率 | 0%（纯文本） | 100%（生成 DDL） | 100% |
| 诊断会话保留 | 仅内存 | 永久存储 | 永久存储 |
| 覆盖的 MySQL 参数分析范围 | ~10 个 | ~50 个 | ~80 个 |
